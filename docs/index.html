User behavior as a predictor for input accuracy¶

Johann Miller, University of Maryland Makeability Lab

Github repo

Introduction¶

Project Sidewalk is an online platform that crowdsources data about sidewalk accessibility. Users navigate streets using Google Street View, and place labels on issues such as crosswalks without curbramps, uneven pavement, and obstacles blocking the path. In order to ensure some level of accuracy in the data, Project Sidewalk has a couple tools at its disposal. Each user has to complete a tutorial before they can begin to report problems. Another option is ground truth seeding, where users place labels in a region that already has established answers. If the user enters data that doesn't match the ground truth, then all of the data they entered can be flagged.

Here, we will investigate another possible option: using the user's interactions with the tool to predict their accuracy. Consider a user who is inactive for long periods, and barely uses any of the tool's features. This user probably gives worse input than a user who works consistently and employs all of the tool's features. Aspects of interaction include mouse movement, keypresses, and others that we can collect while users place labels. If accurate labels correspond to a certain type of usage, then these features could predict the accuracy of a user even in non ground truth regions.

To see if this is possible, we'll use data from ground truth regions to train and test a classifier.

Grading user accuracy¶

We need to rate each user session on its accuracy compared to the ground truth. We do so by counting the number of true positives, false positives, true negatives, and false negatives from the session. These are defined as follows:

True positive¶

The user placed a correct label. Here, the green icon is a label for a curb ramp. The user placed it correctly, so this is a true positive.

False positive¶

The user placed an incorrect label. Here, the user placed a green icon to identify a curb ramp, but none are present.

True negative¶

There was nothing to label, and the user didn't label anything. The amount of empty space necessary to be considered a true negative is the granularity. The scoring was done with three values: 5 meter, 10 meter, and street.

False negative¶

There was something to label, but the user missed it. Here, there is a curb ramp with no label.

Composite scores¶

We can combine the counts of true positives, true negatives, etc. into metrics that tell us about the accuracy of the user.

$\text{accuracy} = \frac{\text{true positives + true negatives}}{\text{true positives + false positives + true negatives + false negatives}}$

$\text{positive predictive value (ppv)} = \frac{\text{true positives}}{\text{true positives + false positives}}$

$\text{negative predictive value (npv)} = \frac{\text{true negatives}}{\text{true negatives + false negatives}}$

$\text{recall} = \frac{\text{true positives}}{\text{true positives + false negatives}}$

$\text{specificity} = \frac{\text{true positives}}{\text{true positives + true negatives}}$

Modeling¶

SVM¶

To model this data, we can use supervised machine learning. One popular class of algorithm is support vector machine (SVM). For classification, SVM selects hyperplanes that minimize error by maximizing the margin between classes of data. For regression, it works similarilty: a hyper plane is fit along the data. If the data is not linear, a kernel function is used to transform the data. Our user accuracy label is continuous, so we will use regression.

Regression¶

SVM regression has two major hyper parameters: $C$, the penalty for error, and $epsilon$, the margin of tolerance for errors. In order to choose values for the hyper parameters, we can test out a range of options. Here, we use the scikit-learn SVM regression implementation with a radial-basis kernel, $epsilon$ of $0.1$, and a range of $C$ values from $0.01$ to $1.0$. The tests are done with 3-fold cross validation.

In [15]:
c_values = np.linspace(0.01, 1, 50)
svr = svm.SVR(epsilon=0.1)
svr_scores = []
svr_err = []
for c in c_values:
    svr.set_params(C=c)
    cv = model_selection.cross_val_score(svr, features, labels.accuracy, cv=3)
    svr_scores.append(np.mean(cv))
    svr_err.append(np.std(cv))
print('Max C:', c_values[np.argmax(svr_scores)])
plt.xlabel('C')
plt.ylabel('cv r^2 score')
plt.errorbar(c_values, svr_scores, label='svr', yerr=svr_err, ecolor='grey')
plt.legend(loc='lower right')
plt.show()

Max C: 0.191836734694

The regressor had the best performance around $C=0.2$ We can rerun a similar test on possible $epsilon$ values.

The regessor had the best performance around $epsilon = 0.05$. We've seen how the parameters impact the model individualy, but not how they interact together. In order to optimize the model with both of them, we can use a grid search to test out each combination of parameters. In addition to the radial-basis function, we also test a polynomial kernel. We also increase the cross validation to 10-fold.

In [17]:
parameters ={
    'C': np.power(10.0,range(-2,3)), 'epsilon': np.power(10.0,range(-4,-1)), 'kernel': ('poly', 'rbf')}

clf = model_selection.GridSearchCV(svr, parameters, n_jobs=-1)

fig = plt.figure()
sns.boxplot(x=scores)
plt.xlabel('cv r^2 score')
fig.set_size_inches(5,1.5)
print("r^2: %0.2f (+/- %0.2f)" % (scores.mean(), scores.std() * 2))

r^2: 0.30 (+/- 0.19)

We received an $r^2$ score of 0.3 with a 95% confidence interval between 0.1 and 0.5. The $r^2$ score is an indication of how well the actual test data fitted the regression. A score of 1.0 would indicate a perfect fit. Our score indicates that the majority of the variation in accuracy is not explained by the model.

Classification¶

Because the SVM is struggling to predict accuracy as a continuous variable, we can ease its task. Using binning, we can discretize accuracy into categories. Here, we halve our data into two categories: users below median accuracy, and users above. To predict which group a user belongs to, we can use an SVM classifier. Here, we score the classifier's predictions using gridsearch on $C$ and kernel type with 10-fold cross validation.

In [18]:
parameters = {'C': np.power(10.0,range(-2,3)), 'kernel': ('poly', 'rbf')}
binned_accuracy = (labels.accuracy < labels.accuracy.median()).apply(lambda x: 1 if x else 0)
svc = svm.SVC()
clf = model_selection.GridSearchCV(svc, parameters, n_jobs=-1)
scores = model_selection.cross_val_score(clf, features, binned_accuracy, cv=10)
fig = plt.figure()
sns.boxplot(x=scores)
plt.xlim(0,1)
fig.set_size_inches(7,1.5)
plt.xlabel('cv score')
print("cv: %0.2f (+/- %0.2f)" % (scores.mean(), scores.std() * 2))

cv: 0.68 (+/- 0.17)

On the binned data, the SVM correctly classifies about 70% of the users. Since the data is evenly distributed between binary categories, a random classification would result in 50% being correct. To understand where the classifier makes mistakes, we can view the confusion matrix.

Random Forest¶

Random forest is another modeling algorithm. It works by creating many decision trees based on the features and binning all their results into a single prediction. The two major hyperparameters are the number of trees, and the maximum number of features that it will try at each decision point. Here, we will fit a regressor to predict accuracy and a classifier to predict between the low and high accuracy groups.

Naive Bayes classification¶

Bayes classification is a technique based on the Bayesian theorem. One of its assumptions is that all features of the are independent. We know that is not the case, so we must remove features with high pairwise correlations. The model takes a prior of the distribution between classes. Here, the distribution will be assumed from the training set.

In [26]:
# remove pairwise correlated features
cor_features = set()
for i, row in features.corr().iterrows():
    if i not in cor_features:
        cor_features = cor_features.union(set(((row[row > 0.9]).index)))
        cor_features -= set([i]) # not correlated with self
uncor_features = set(range(len(feature_cols))) - cor_features
bayes_features = features[list(uncor_features)]
print(len(cor_features), 'features removed')

# train and score model
gnb = naive_bayes.GaussianNB()

Conclusion¶

We built models for user accuracy using SVM, Random Forest, and Bayes classification. First, we attempted to predict an exact number for the accuracy of users. Random forest regression was able to explain between 26% and 72% (95% confidence) of the variance in accuracy using the features. Next, we tried to make the problem easier by binning users into low and high accuracy groups. Here, the random forest classifier correctly picks the group for between 55% and 97% (95% confidence) of the users. For comparison, a random selection between the two classes would result in 50% of classifications being correct. SVM classification performed similarly, with a 95% confidence interval between 51% and 85% accuracy. Bayes classification performed the worst, with a 95% confidence interval between 49% and 71% accuracy.

The large confidence intervals on these estimates are likely due to a lack of data. We had 308 samples where each sample had 291 features (208 after removing features with pairwise correlation). Since the feature space is large, two subsets of the data could be very different. This leads to uncertainty about the quality of a model. If more data is added, the groups will become more uniform and the prediction quality may become more consistent. Still, we have shown that data about user behavior does provide some amount of predictive power about the accuracy of the data they enter.

In application to Project Sidewalk, this model could serve as a preliminary moderator for the crowd sourced data. The model wouldn't remove data on its own, since the range of its possible accuracy is wide. However, since the model is completely autonomous, it could be a part of a pipeline where the model flags possible low quality data that is verified using a more time-expensive operation.

Comparison to ground truth seeding¶

Ground truth seeding is a standard method for validating crowdsourced data. Our predictive approach is not meant to replace it, but rather to fill in gaps where it may fail. For one, ground truth seeding relies on the assumption that users behave consistently. If a user varies wildly, then they may pass the ground truth region while going unnoticed with bad data elsewhere. The predictive approach may have better results here since it can monitor all of the data that a user enters. Another risk of ground truth seeding is that users may learn which regions are the control, especailly if the amount of ground truth data is limited. If a user recognizes a region as a ground truth, then he or she may change behavior for that region only. Again, the predictive method has the advantage of accessing all of a user's behavior. Finally, using ground truth seeding has a cost on the total amount of data that can be collected. Assuming that a user only enters accurate data, any amount of time devoted to ground truth regions is lost time for collecting new data. Meanwhile, the predictive method does not take time away from contributing users.

Wider use¶

Crowdsourcing has risen in popularity in recent years, with platforms like Amazon Mechanical Turk allowing tasks to be outsourced to an arbitrary number of people. Our current model is completely specific to Project Sidewalk. However, it is possible that a similar model could be trained on data from any service that keeps user logs and has a scoring metric on its users. Possible applications would be on a forum, where users may behave differently while positing an insult, or forms, where users may act differently when lying. Such an approach may run into problems if users know it is being used. As mentioned previously, ground truth seeding can suffer if users become aware and change their behavior. Similarily, users could recognize what types of actions are flagged by the model, and change their behavior accordingly. More research will be necessary to determine how this risk can be minimized, and to know the general effectiveness of this approach outside of the environment we tested here.
